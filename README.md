# ING Toolkit

A set of helpers to interact with files generated by ING bank.

**ING Group N.V.** (Dutch: *ING Groep*) is a Dutch multinational banking and financial services corporation headquartered in Amsterdam:

- https://en.wikipedia.org/wiki/ING_Group
- https://ing.ro

## Installation

Package Manager:

```
Install-Package DustInTheWind.Ing.Toolkit
```

.NET CLI:

```
dotnet add package DustInTheWind.Ing.Toolkit
```

## Runtime Requirements

- Library target framework: `.NET 8.0` (`net8.0`)

## Quick Start

### Step 1 - Export Statement CSV File

- Open the ING Homebank web application in your browser (https://www.homebank.ro).

- In the "Portofoliu" tab, make sure the current account box is expanded.
- Click the three dots "Mai multe" and select "Extras de cont" (Statement)
- Choose the month and export the statement as CSV.

### Step 2 - Parse the Exported Document

The `StatemewntDocument` class provides several methods to parse and load a CSV file, to accommodate different input sources. Here it is one of them:

**From a file path:**

```csharp
StatementDocument document = StatementDocument.LoadFile("statement.csv");

foreach (BankTransaction transaction in document)
{
    ...
}
```

## CSV File Structure

**Important**: the file exported by the ING Homebank and saved with a `.csv` extension is not a plain/comma-separated CSV file in the traditional sense. It is an export produced by Excel and represents the printable/exportable statement as one workbook with multiple pages.

- The document is organized in multiple pages. Each page corresponds to a printable page in the original exported file.
- Each page begins with a page header that contains identifying information for the account holder: the full name, the CNP (personal numeric code) and the address.
- Each page ends with a small signature section (a printed area reserved for signatures).
- The final page contains additional summary information: opening balance, closing balance, totals and other aggregated values for the entire statement.

### CSV Inconsistencies 

- There is not a single continuous table. The document is a sequence of pages.
- The CSV file contains multiple empty columns inserted between the ones with data.

- The header cells are shifted one column to the right compared to the columns containing the data (e.g. `Data` header is in column 2 while the dates are placed in column 0)
- Each bank transaction spans across multiple csv records. First record has all values, but an incomplete description. Subsequent description lines are provided as separate csv records that have no other data but the description column.
- etc...

### Columns and mapping to `BankTransaction`

The parser maps the important columns to the `BankTransaction` model as follows:

- **Data** (Date)
  - Mapped to `BankTransaction.Date` (`DateOnly`).
  - Date format: `dd MMM yyyy` and parsed using the Romanian culture (`ro-RO`).
- **Detalii tranzactie** (Details)
  - Mapped to `BankTransaction.Details` (`List<string>`).
  - The description can span multiple printable lines. When a transaction's detail continues on the next CSV row the parser appends that row's detail cell to the `Details` list. Continuation rows are detected by an empty first cell (no date) and no page footer marker.
- **Debit** (Debit amount)
  - Mapped to `BankTransaction.DebitAmount` (`decimal?`).
  - Parsed using the Romanian number format (comma decimal separator). Empty cells are treated as `null`.
- **Credit amount** (Credit)
  - Mapped to `BankTransaction.CreditAmount` (`decimal?`).
  - Parsed the same way as the debit amount.
